# 00. 개요, 배경, 포지셔닝

## 0.1 한 줄 정의

> PaperTrail은 **JSON 데이터를 받아 계약서, 통지서, 확인서, 청구서를 생성하고, 어떤 데이터와 템플릿으로 만들어졌는지 추적**하는 버전 관리형 전자문서 생성 및 증적 플랫폼이다.

## 0.2 해결하려는 문제

사내에 여러 업무 시스템이 있고, 각 시스템이 저마다 PDF를 만든다. 이 방식은 다음 문제를 낳는다.

1. **중복 비용** — 시스템마다 PDF 라이브러리, 폰트, 렌더링 CPU/메모리 리소스를 각자 부담한다.
2. **품질 편차** — 팀마다 다른 라이브러리, 레이아웃 규칙을 써서 문서 품질/브랜드가 제각각이다.
3. **증적 부재** — "이 PDF가 정확히 어떤 데이터와 어떤 템플릿 버전으로 생성됐는가?"에 답하지 못한다. 공공/금융/계약 문서에서 이는 곧 리스크다.
4. **대량 처리 취약** — 통지서, 고지서처럼 수만 건을 한 번에 생성할 때 재시도, 멱등성, 장애 복구가 각 시스템에 흩어져 있다.

## 0.3 해결 방식 — 공용 문서 렌더링 SaaS

문서 생성 기능을 **별도의 공용 서비스로 분리**하여 내부 SaaS로 운영한다.

- 각 사내 서비스는 **JSON만 전송**한다. PDF 라이브러리/리소스를 직접 부담하지 않는다.
- 템플릿은 **한 번 등록(publish)** 하고, 어디서든 참조(reference)로 렌더링한다.
- 모든 렌더는 **입력, 출력, 템플릿의 해시와 함께 기록**되어 완전한 감사 추적과 재현성을 제공한다.

```
┌───────────────┐   ┌───────────────┐   ┌───────────────┐
│  HR 시스템    │   │  민원 시스템  │   │  정산 시스템  │
└───────┬───────┘   └───────┬───────┘   └───────┬───────┘
        │  JSON             │  JSON             │  JSON
        └───────────────────┼───────────────────┘
                            ▼
                 ┌────────────────────┐
                 │     PaperTrail     │  ← 공용 문서 렌더링 SaaS
                 │  (버전관리+증적)   │
                 └────────────────────┘
```

## 0.4 왜 Papermake 위에 짓는가

[Papermake](https://github.com/rkstgr/papermake)는 "문서 템플릿을 위한 Docker 레지스트리"를 표방하는 Rust 프로젝트로, PaperTrail의 요구와 정확히 맞는 특성을 이미 갖고 있다.

| Papermake 특성 | PaperTrail에서의 의미 |
|----------------|----------------------|
| **SHA-256 콘텐츠 주소 저장** | 템플릿을 해시로 고정 → 불변 버전 보장 |
| **불변 매니페스트 + 가변 태그** | `invoice:v1.0.0`(고정) vs `invoice:latest`(이동) 구분 |
| **렌더 입력/출력 해시 기록** | 증적, 재현성의 근간 |
| **PDF/A-2b, PDF/A-3b 지원** | 전자문서 장기보관/전자세금계산서 대응 |
| **S3 + ClickHouse 기반** | PaperTrail의 저장 아키텍처와 동일 |
| **Typst 렌더링 엔진** | LaTeX 대비 빠르고 현대적인 템플릿 언어 |

> **중요:** Papermake의 `papermake-worker`는 현재 **실질적인 큐 처리 로직이 없는 초기 스텁** 상태다. 따라서 비동기 대량 처리, 재시도, DLQ, 동시성 제어, 멱등성은 **PaperTrail(NestJS 계층)에서 직접 설계**한다. 이것이 이 프로젝트의 핵심 엔지니어링 기여 지점이다.

## 0.5 설계 원칙

1. **Papermake를 그대로 외부에 노출하지 않는다.** NestJS가 앞단을 감싸 인증, 멀티테넌트, 멱등성, Webhook, Signed URL을 담당한다.
2. **동기 요청 안에서 렌더를 끝내지 않는다.** 모든 생성 요청은 작업 큐로 분리한다.
3. **증적은 부가 기능이 아니라 1급 기능이다.** 모든 문서는 재현 가능한 형태로 기록된다.
4. **개인정보(PII)는 최소 저장, 마스킹, 암호화한다.** 렌더 입력 원문을 분석 스토어에 그대로 쌓지 않는다.
5. **멱등성으로 중복 생성을 막는다.** 재시도/네트워크 오류에도 문서는 정확히 한 번만 생성된다.

## 0.6 범위 밖 (Non-Goals) — MVP 기준

- 문서 전자서명(PKI/공인인증) 자체 구현 — 후속 단계 또는 외부 연동
- WYSIWYG 템플릿 에디터 — 초기에는 Typst 소스 + 미리보기까지만
- 결제/과금 시스템 완성 — 스키마만 준비, 실제 빌링은 후순위
- 카카오, 네이버 전자문서 발송 — 차별화 로드맵 항목(후속)

## 0.7 용어 정의

| 용어 | 정의 |
|------|------|
| **Tenant(테넌트)** | PaperTrail을 이용하는 조직/서비스 단위. 데이터 격리의 기본 경계 |
| **Template(템플릿)** | Typst 소스 + JSON Schema로 구성된 문서 양식 |
| **Tag(태그)** | 특정 템플릿 매니페스트를 가리키는 가변 포인터 (`latest`, `staging`) |
| **Reference(참조)** | 렌더 시 지정하는 `name:tag` 또는 `name@sha256:...` |
| **Manifest(매니페스트)** | 파일명 → 콘텐츠 해시 매핑. 불변 |
| **Render(렌더)** | 템플릿 + 데이터 → PDF 생성 1회 |
| **Document(문서)** | 렌더 결과물 + 그 증적 메타데이터의 논리 단위 |
| **Idempotency Key(멱등키)** | 동일 요청의 중복 생성을 막는 클라이언트 제공 키 |
| **DLQ** | Dead Letter Queue. 최대 재시도 후에도 실패한 작업의 격리 큐 |
| **Audit Trail(증적)** | 문서가 언제, 무엇으로, 어떻게 생성됐는지의 기록 사슬 |

# 06. 로드맵, 마일스톤

## 6.1 최종 추천 범위

처음에는 아래 4대 흐름만 완성한다.

```
템플릿 등록
  → JSON 데이터 입력
  → 비동기 PDF 생성
  → 생성 이력 및 PDF 다운로드
```

그다음 차별화 기능을 추가한다.

```
CSV 대량 생성
Webhook
PDF/A
문서 재현
템플릿 승인
카카오, 네이버 전자문서 발송 연동
```

## 6.3 우선순위 원칙

1. **핵심 차별화(증적/재현/멱등)를 M1에서 먼저 증명.** 단순 렌더는 Papermake가 이미 함.
2. **큐/워커는 추상화**해서 로컬(BullMQ)과 클라우드(SQS)를 같은 코드로.
3. **보안/멀티테넌트는 처음부터** 미들웨어에 심는다(나중 이식은 비용 큼).
4. Admin UI는 API가 안정된 뒤 M3에서 집중.

## 6.4 리스크 / 열린 질문

| 항목                 | 내용                                                                     |
| -------------------- | ------------------------------------------------------------------------ |
| Papermake API 안정성 | 스텁/실험 단계 기능이 있어 버전 pin + 통합 테스트 필요                   |
| 재현성 정밀도        | 폰트/렌더러 버전 차이로 `outputHash`가 흔들릴 수 있음 → 렌더러 버전 고정 |
| 입력 정규화 규칙     | `inputHash` 안정성의 핵심. 초기에 확정하고 문서화                        |
| 큐 전환 비용         | BullMQ ↔ SQS 시맨틱 차이(가시성 타임아웃 등) 흡수                        |
| PII 정책             | 테넌트별 보관/삭제 요구가 다를 수 있음 → 정책 파라미터화                 |
